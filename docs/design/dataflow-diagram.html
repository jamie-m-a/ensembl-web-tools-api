End-to-end dataflow after the spec-driven changes

How the frontend, backend, and the new API exchange options, config, parsing and display across one submission. The new API is drawn as a distinct service but is realised today as a local JSON file. Order and endpoints verified against the current code; reflects the decisions locked in review.

Sequence — one submission, start to finishdev / prod branches tagged inline

Frontend owns

  • Input form + results UI via generic renderers (+ small custom kit: view-in-app popup, show-more)
  • Requests the input form from the backend on species selection
  • Triggers submission; polls status every 15s
  • Requests filtered pages + downloads

New API owns

  • Endpoint 1 — options + input-render spec, keyed on species
  • Endpoint 2 — config + parsing spec + display spec, for the selected options
  • Pure data: no logic, backend-authoritative for parsing

Backend owns

  • Relays endpoint 1 and applies activation (get_visible_panels)
  • Always-on base config; merges + emits config.ini
  • Pins parsing + display spec per job
  • Both checks; launches/polls the pipeline; parses; filters + downloads

Pipeline

  • Prod — Nextflow via Seqera; output mounted for the backend
  • Dev — manual HPC run; output hand-placed in dev-data
  • Adds required headers on rerun when they're missing

Decisions locked in this review

DecidedEndpoint 1 stays relayed; activation stays on the backend

The frontend may not hold the genome metadata needed to resolve options for every species (GRCh38 is easy, others get more complex), so the backend fetches endpoint 1 and runs activation via get_visible_panels.

ConfirmedEndpoint 1 fires on species selection

Kept as the current trigger — no deferral to data entry.

ConfirmedEndpoint 2 is options-aware

It receives the selected options and returns config, parsing and display for those options — which is what makes the per-job header check meaningful.

ConfirmedResults display is pinned to the submitted options

The pinned display spec is used at results time rather than re-fetching live panels — cleaner, and it fixes a latent pinning gap.

DecidedMissing headers: pipeline adds them on rerun, capped at 3

The required headers are needed by the parser and shouldn't go missing often. On a miss, prod reruns the pipeline (which adds them), max 3 retries; dev only warns.
